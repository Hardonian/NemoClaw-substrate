# Packaging Summary

This document describes the deployment bundles, Docker hardening notes, and offline demo instructions for NemoClaw.

## Deployment Bundles

### CLI Package

The NemoClaw CLI is distributed as an npm package with the following structure:

| Path | Purpose |
|------|---------|
| `bin/` | CLI launcher scripts (CommonJS) |
| `dist/` | Compiled CLI output (JavaScript) |
| `src/` | TypeScript source (development) |
| `package.json` | Package metadata and dependencies |

### Plugin Package

The NemoClaw plugin is a separate npm project in the `nemoclaw/` subdirectory:

| Path | Purpose |
|------|---------|
| `nemoclaw/src/` | Plugin TypeScript source |
| `nemoclaw/dist/` | Compiled plugin output |
| `nemoclaw/package.json` | Plugin package metadata |

### Blueprint Package

The blueprint defines the sandbox configuration and network policies:

| Path | Purpose |
|------|---------|
| `nemoclaw-blueprint/` | Blueprint YAML manifests |
| `nemoclaw-blueprint/policies/` | Network policy presets |
| `nemoclaw-blueprint/model-specific-setup/` | Agent compatibility registry |

### Agent Skills

Generated user skills are packaged under `.agents/skills/`:

| Package | Purpose |
|---------|---------|
| `nemoclaw-user-*` | End user skills (autogenerated from docs) |
| `nemoclaw-maintainer-*` | Project maintainer skills |
| `nemoclaw-contributor-*` | Codebase contributor skills |

## Docker Hardening Notes

### OpenShell Container

NemoClaw runs inside OpenShell containers with the following security posture:

| Control | Setting | Purpose |
|---------|---------|---------|
| Capability drops | All non-essential capabilities dropped | Minimize container privileges |
| Process limits | Configured via OpenShell | Prevent resource exhaustion |
| Network policies | Applied at container level | Control egress traffic |
| SSRF validation | Enabled in blueprint | Prevent server-side request forgery |
| Credential sanitization | Active on all exports | Prevent secret leakage |

### Container Image Security

- Base images are pulled from official sources.
- Container images are rebuilt when OpenShell updates.
- No secrets are baked into container images.

### Sandbox Isolation

Each sandbox runs in an isolated OpenShell container with:

- Separate workspace directory.
- Independent network policy enforcement.
- Isolated credential store.
- No shared state between sandboxes.

## Installation Bundles

### Quick Install

The recommended installation method:

```console
$ curl -fsSL https://www.nvidia.com/nemoclaw.sh | bash
```

### Manual Install

For environments without internet access or when full control is required:

```console
$ git clone https://github.com/NVIDIA/NemoClaw
$ cd NemoClaw
$ npm install
$ npm link
$ cd nemoclaw && npm install && npm run build && cd ..
$ cd nemoclaw-blueprint && uv sync && cd ..
```

## Offline Demo Instructions

### Prerequisites for Offline Demo

- Node.js v22.16+ installed.
- Docker installed and running.
- OpenShell runtime installed.
- NemoClaw repository cloned with all dependencies.

### Steps for Offline Demo

1. Ensure all dependencies are installed before going offline.
2. Configure inference for local inference provider (no network required):

   ```console
   $ nemoclaw onboard
   # Select local inference provider during setup
   ```

3. Launch the sandbox:

   ```console
   $ nemoclaw start
   ```

4. Verify sandbox operation:

   ```console
   $ nemoclaw status
   $ nemoclaw diagnostics
   ```

5. Demonstrate network policy controls:

   ```console
   $ nemoclaw policy list
   ```

6. Demonstrate snapshot and rollback:

   ```console
   $ nemoclaw config-snapshot --pre-upgrade
   ```

### Offline Limitations

| Feature | Available Offline | Notes |
|---------|-------------------|-------|
| Sandbox launch | Yes | Requires pre-installed OpenShell |
| Local inference | Yes | Requires local model |
| Cloud inference | No | Requires network access |
| Policy updates | No | Requires network for policy presets |
| Dependency install | No | Requires npm registry access |
